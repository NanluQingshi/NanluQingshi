## 二、模型层：你需要知道的

### 2.1 大语言模型（LLM）

LLM 是一切的基础，你不需要训练它，但要知道怎么选。

| 模型 | 厂商 | 特点 | 适合场景 |
|------|------|------|----------|
| Claude 3.x / 4.x | Anthropic | 逻辑强、长上下文、安全性好 | 复杂推理、代码、长文档 |
| GPT-4o | OpenAI | 综合能力强、多模态 | 通用场景 |
| 通义千问 / 文心 / 混元 | 国内厂商 | 中文理解好、合规方便 | 国内部署、中文业务 |
| Llama 3 / Qwen | Meta / 阿里开源 | 可私有化部署 | 数据敏感场景（财务！）|
| 小模型（GLM-4-Flash 等） | 各厂 | 速度快、成本低 | 简单分类、路由任务 |

**财务场景关键考虑**：数据敏感 → 优先考虑私有化部署的开源模型，或支持私有化的商业模型。

### 2.2 多模态模型

不只处理文字，还能处理图片、PDF、语音。

| 能力 | 用途 | 财务场景 |
|------|------|----------|
| 图片理解 | 识别图表、截图 | 发票图片识别、单据扫描 |
| PDF 解析 | 读取文档结构 | 合同解析、财报分析 |
| 语音转文字 | 会议记录 | 财务会议纪要自动化 |

## 三、工程层：你的核心技能

### 3.1 Prompt Engineering（提示词工程）

**是什么**：通过设计给 AI 的"指令"来控制输出质量。

**核心技巧**：

| 技巧 | 说明 | 示例 |
|------|------|------|
| 角色设定 | 告诉 AI 它是谁 | "你是一个专业的财务审计助手" |
| 少样本示例（Few-shot）| 给几个例子让 AI 学格式 | 给 3 条发票，让 AI 按格式输出第 4 条 |
| 思维链（CoT）| 让 AI 一步步推理 | "请逐步分析这笔账目的合规性" |
| 结构化输出 | 要求输出 JSON/表格 | "以 JSON 格式返回，字段为：金额、类型、日期" |
| 系统提示 | 固定背景和约束 | 注入公司财务规则、会计准则 |

**财务场景关键**：系统提示里注入会计准则（如企业会计准则 2024）、公司报销规则，能大幅提升准确率。

---

### 3.2 RAG（检索增强生成）

**是什么**：让 AI 在回答前先去"查资料"，解决 LLM 知识截止和幻觉问题。

**工作流程**：
```
用户提问
    │
    ▼
① Embedding：把问题转成向量
    │
    ▼
② 向量检索：从知识库找最相关的文档片段
    │
    ▼
③ 拼上下文：把检索结果 + 问题一起发给 LLM
    │
    ▼
④ LLM 生成答案（基于检索内容，不瞎编）
```

**财务场景应用**：
- 财务政策问答（把公司报销制度、税务规则存入知识库）
- 合同条款查询（合同文本向量化后语义检索）
- 财报分析（上传财报 PDF，对话式分析）

**Java 技术栈实现**：
- **Spring AI**：内置 RAG 流水线支持
- **LangChain4j**：`EmbeddingStore` + `RetrievalAugmentor`
- **向量数据库**：pgvector（PostgreSQL 插件）、Milvus、Weaviate

---

### 3.3 Tool Use / Function Calling（工具调用）

**是什么**：让 AI 能调用真实的函数/接口，而不只是生成文字。

**工作流程**：
```
用户："帮我查一下上个月的报销总额"
    │
    ▼
AI 判断需要查数据库 → 生成工具调用请求
    { "tool": "queryExpense", "params": { "month": "2026-04" } }
    │
    ▼
你的代码执行这个函数，查数据库
    │
    ▼
把结果返回给 AI
    │
    ▼
AI 用结果组织自然语言回答用户
```

**这是 Claude Code 的核心机制，也是财务 AI 应用的核心。**

**财务场景工具库（你们需要封装的）**：

| 工具名 | 功能 |
|--------|------|
| `queryExpense` | 查报销数据 |
| `queryInvoice` | 查发票信息 |
| `createApproval` | 发起审批流 |
| `generateReport` | 生成报表 |
| `checkBudget` | 查预算余额 |
| `validateTaxNo` | 校验税号合规性 |

---

### 3.4 Agent（智能体）

**是什么**：能自主规划、多步执行、使用工具的 AI 系统。比单次问答更复杂。

**和普通问答的区别**：

```
普通问答：用户问 → AI 答（一来一回）

Agent：用户说目标 → AI 自主规划步骤 → 循环执行工具 → 完成目标
```

**Agent 的几种模式**：

| 模式 | 说明 | 财务场景 |
|------|------|----------|
| ReAct | 思考→行动→观察→循环 | 自动对账（查数据→发现差异→追溯原因）|
| Plan & Execute | 先制定计划，再逐步执行 | 月结流程自动化 |
| Multi-Agent | 多个 AI 分工协作 | 一个负责数据收集，一个负责审计，一个负责报告 |

**Java 实现**：
- **Spring AI**：`ChatClient` + Tool 注解，简单 Agent
- **LangChain4j**：`AiServices` + `@Tool` 注解，功能更全

---

### 3.5 工作流编排（非纯 Agent 场景）

**是什么**：把 AI 嵌入固定的业务流程，不完全自主，更可控。

**适合财务场景**（财务操作出错代价大，需要确定性）：

```
发票处理工作流示例：

上传发票图片
    │
    ▼
OCR + 多模态模型提取字段（发票号、金额、税率）
    │
    ▼
规则引擎校验（金额是否超标、税号是否合规）
    │
    ▼
AI 判断风险等级（低风险自动通过/高风险人工审核）
    │
    ▼
写入系统 + 触发审批流
```

**工具**：Dify、Coze（低代码编排）；或自己用 Spring AI 写流水线。

## 四、基础层：了解即可

### 4.1 Embedding（向量化）

把文字转成一串数字（向量），让计算机能计算"语义相似度"。RAG 的基础。

- 常用模型：`text-embedding-3-small`（OpenAI）、`bge-m3`（开源中文效果好）
- 你只需要会调 API，不需要训练

### 4.2 向量数据库

专门存储和检索向量的数据库。

| 数据库 | 特点 | 推荐场景 |
|--------|------|----------|
| pgvector | PostgreSQL 插件，成本低 | 数据量不大，已有 PG |
| Milvus | 开源，高性能，可私有化 | 大规模、对性能有要求 |
| Chroma | 轻量，开发调试方便 | 本地开发、原型验证 |

### 4.3 模型微调（Fine-tuning）

用你的数据进一步训练模型，让它更懂你的业务。

- **什么时候需要**：通用模型在你的领域表现不好时（如专业财务术语识别差）
- **轻量方案**：LoRA/QLoRA，不需要全量训练
- **财务场景**：大多数情况 RAG + 好的 Prompt 就够了，微调是进阶选项

## 五、财务 AI 应用场景全景

### 5.1 场景地图

```
财务 AI 应用场景

├── 【智能问答类】（RAG 为主）
│   ├── 财务政策/报销规则问答
│   ├── 税务法规查询
│   └── 合同条款解读
│
├── 【单据处理类】（多模态 + 结构化提取）
│   ├── 发票 OCR + 信息提取
│   ├── 费用报销单自动填写
│   ├── 合同关键信息提取
│   └── 银行流水分类标记
│
├── 【数据分析类】（Text-to-SQL + 可视化）
│   ├── 自然语言查报表（"上季度各部门费用对比"）
│   ├── 财报自动解读
│   ├── 预算执行分析
│   └── 异常数据预警
│
├── 【流程自动化类】（Agent 为主）
│   ├── 自动对账（银行流水 vs 系统记录）
│   ├── 月结流程编排
│   ├── 审批流智能路由
│   └── 催款提醒自动化
│
└── 【风控合规类】（分类 + 规则 + AI）
    ├── 异常交易检测
    ├── 报销合规审查
    └── 税务风险提示
```

### 5.2 按落地难度排序

| 难度 | 场景 | 技术 | 理由 |
|------|------|------|------|
| ★ 简单 | 财务政策问答 | RAG | 只读不写，风险低 |
| ★★ | 发票信息提取 | 多模态 LLM | 输出结构化，易校验 |
| ★★ | 自然语言查报表 | Text-to-SQL | 只读数据库，安全 |
| ★★★ | 报销合规审查 | LLM + 规则引擎 | 需要准确率高 |
| ★★★ | 自动对账 | Agent + 工具调用 | 多步骤，需兜底 |
| ★★★★ | 月结流程自动化 | Multi-Agent | 涉及写操作，高风险 |

**建议**：先做"只读"场景（问答、查询、分析），再做"写操作"场景（审批、对账、记账），后者必须加人工确认环节。

## 六、Java 技术选型建议

### 6.1 主框架选择

| 框架 | 推荐度 | 说明 |
|------|--------|------|
| **Spring AI** | ★★★★★ | Spring 官方出品，和 Spring Boot 无缝集成，首选 |
| **LangChain4j** | ★★★★ | 功能最全，Agent/RAG 支持好，Spring AI 不够用时用这个 |

两者可以混用，Spring AI 做主框架，LangChain4j 补充高级功能。

### 6.2 典型技术栈

```
财务 AI 应用后端技术栈

Spring Boot 3.x
    ├── Spring AI（LLM 调用、RAG、Tool Use）
    ├── LangChain4j（复杂 Agent 场景）
    ├── MyBatis-Plus（业务数据库）
    ├── pgvector / Milvus（向量存储）
    ├── Redis（缓存、会话）
    └── RabbitMQ（异步任务，如批量文件处理）

模型选择
    ├── 外网可用：Claude API / OpenAI API
    └── 私有化部署：Ollama + Qwen / Llama（财务数据保密）
```

## 七、学习路径建议（结合你的背景）

### 第一步：能跑通一个完整链路（1-2 周）

```
① 注册 Claude API / OpenAI API
② 用 Spring AI 写一个流式对话接口（后端）
③ 前端接上，能看到打字机效果
④ 加一个工具：让 AI 能查你的数据库
```

### 第二步：做一个 RAG 原型（2-3 周）

```
① 把公司报销制度 PDF 切片存入 pgvector
② 用户提问 → 向量检索 → 喂给 LLM → 回答
③ 验证准确率，调整切片策略和 Prompt
```

### 第三步：做一个完整的财务 AI 功能（1 个月）

选其中一个做完整：
- 发票信息提取 + 合规校验
- 自然语言查财务报表（Text-to-SQL）
- 财务政策知识库问答系统

## 八、关键概念速查

| 术语 | 一句话解释 |
|------|-----------|
| LLM | 大语言模型，AI 的大脑 |
| Prompt | 给 AI 的指令/上下文 |
| Token | LLM 处理文本的基本单位，影响费用和长度 |
| Context Window | LLM 一次能看多长的文本（如 200K tokens）|
| Embedding | 把文字变成向量数字 |
| RAG | 检索增强生成，让 AI 先查资料再回答 |
| Tool Use | 让 AI 能调用函数/接口 |
| Agent | 能自主规划和多步执行的 AI |
| Fine-tuning | 用自己数据进一步训练模型 |
| Hallucination | AI 一本正经地编造不存在的信息（幻觉）|
| System Prompt | 对话开始前给 AI 的固定背景指令 |
| Streaming | 流式输出，AI 打字机效果的原理 |
| MCP | Model Context Protocol，AI 接入外部工具的标准协议 |
